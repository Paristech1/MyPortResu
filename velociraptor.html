Velociraptor DFIR Lab — Endpoint Monitoring

Stood up an open-source DFIR collection platform from scratch and used it to remotely monitor and investigate a live Windows endpoint — replicating the core triage workflow a SOC or incident-response team runs against a compromised machine. A Velociraptor server on Kali Linux, a Windows 11 agent reporting in over TLS, and live forensic collection of filesystem, registry, and network artifacts from a central console.

Full Project Journal ↗

Lab Overview

A two-node investigation lab running entirely in virtual machines on Apple Silicon (UTM). The goal: deploy endpoint visibility, then investigate — exactly what an analyst does when a security alert fires and there's no time to physically image every hard drive. Based on Sam Bowne's CNIT 152 lab "IR 371: Velociraptor Server on Linux."

  • Velociraptor server: Kali Linux (ARM64) hosting the collection server and web console.
  • Monitored endpoint: Windows 11 (x86_64) running the Velociraptor agent, reporting in over TLS.
  • Network: both VMs on a shared virtual network so the agent could reach the server frontend.

Velociraptor is an open-source DFIR and threat-hunting platform (maintained by Rapid7). A lightweight agent runs on endpoints; a central server lets investigators query them using VQL (Velociraptor Query Language) — a SQL-like language that interrogates a machine's live state. Reusable VQL bundles are artifacts; running one across a whole fleet at once is a hunt. It's often described as an open-source alternative to commercial EDR, leaning toward investigation and hunting rather than automated real-time blocking.

Building the Platform

  • Deployed the Velociraptor server (v0.77.1) on Linux — downloaded the binary, generated the server configuration, and bound the web GUI and client frontend to the host's network IP so the endpoint could connect.
  • Configured secure TLS client-server communications and created an administrator account for the console.
  • Packaged a Windows agent (MSI) using the server's built-in artifact, served it over HTTP, and installed it on the Windows 11 endpoint.
  • Confirmed the agent checked in and registered in the console — establishing live, two-way endpoint visibility.

Setup note: because the host is Apple Silicon, the arm64 server binary was used (the project assumes amd64), and the GUI bind_address had to be changed from 127.0.0.1 to 0.0.0.0 so the endpoint on the virtual network could reach the console.

Investigating the Endpoint

With the agent live, I used the console to remotely investigate the endpoint without disturbing the evidence — browsing the disk, registry, and network state through the agent rather than touching the live OS. The lab is graded by five "flags," each proving a different investigation capability:

  • 371.1 — Server Version: read the build/version from the console "Server status" dashboard, confirming a working management plane.
  • 371.2 — Agent Name: the Windows agent (DESKTOP-EL6QVN3) checked in and registered — live, two-way visibility over TLS.
  • 371.3 — Registry: remotely read the endpoint's live Windows registry (HKEY_CURRENT_USER) through the Virtual File System — a core forensic source for persistence and user activity.
  • 371.4 — File System: raw NTFS browsing of the remote disk (\\.\C: → Users → Default); the agent parses the filesystem directly, seeing files the live OS would hide.
  • 371.5 — DestPort / RPort: collected a live network artifact and identified the agent's own outbound beacon back to the server.

Reading the Network Artifact

Collecting a netstat artifact and filtering it surfaced the established TCP session from the Windows endpoint back to the Kali server — the agent's own beacon to the frontend on port 8000:

  • "Name": "Velociraptor.exe", "Status": "ESTAB"
  • "Laddr.IP": "192.168.64.51" — the Windows endpoint
  • "Raddr.IP": "192.168.64.8" — the Kali server
  • "Raddr.Port": 8000 — the flag: the channel built earlier in the lab

Troubleshooting — The Real DFIR Work

  • VFS "No data available": a refresh initially returned nothing because the client had briefly dropped offline. The VFS only works against a live, connected client — once it showed green Connected, refreshing populated the tree.
  • Hung enrichment artifact: the project's Windows.Network.NetstatEnriched artifact repeatedly stalled (state stuck on "running", 0 rows) on the emulated Windows VM. I verified the service account with sc qc velociraptor, ruled out a privilege issue, then switched to the plain Windows.Network.Netstat artifact, which completed instantly (47 rows).
  • Bonus observation: the same netstat dump revealed a splunkd.exe process beaconing out on 9997 — a Splunk forwarder shipping logs, tying back to my Splunk SIEM lab.

Why It Matters

When a security alert fires, responders rarely have time to physically image every hard drive. Tools like Velociraptor let an analyst query endpoints remotely and at scale — pulling processes, registry keys, files, and network state from a central console in minutes. This lab is that workflow in miniature: deploy visibility, then investigate. It maps directly to the day-to-day of a junior SOC / DFIR analyst.
