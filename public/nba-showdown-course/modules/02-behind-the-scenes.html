Behind the Scenes

We know how the showdown works on screen, but where do the players come from? They don't live in our code. Instead, we have to go out and fetch the raw box scores from the real world.

The Three-Tier System

Our app consists of a frontend (the interface you click on), a backend (our Python brain that shapes the data), and a third-party API (the official NBA data servers). They have to work together perfectly.

The Relay Race

When you start a fresh game, your browser doesn't talk directly to the NBA. It talks to our backend server, which acts like a specialized courier.

The Need for Speed

On an average night, there might be 10 NBA games playing. If the backend asked the NBA for Game 1's boxscore, waited for the reply, then asked for Game 2, and so on, you would be staring at a loading spinner forever. It's like a supermarket only having one checkout line open.

To fix this, the backend uses concurrency. It asks for all the games simultaneously by spawning "workers", like opening 5 checkout lanes at once.

CODE 
with ThreadPoolExecutor(max_workers=5) as executor:
    futures = {executor.submit(_fetch_box_score, gid): gid for gid in game_ids}
    for future in as_completed(futures):
        try:
            player_pool.extend(future.result())
        except Exception as e:
            print(f"Error in thread for {futures[future]}: {e}")
PLAIN ENGLISH

Spin up a pool of 5 workers who can fetch requests simultaneously.

Give each worker an assignment: "Go fetch the box score for a specific game ID!"

As soon as *any* worker returns from the NBA with their results...


Take the players they found and dump them into our giant pool of players.

But if a worker crashes or gets an error instead...

Safely ignore that broken game so the rest of the app doesn't crash!
