Splunk Project

A hands-on blue-team home lab built from bare metal up: a Windows Active Directory domain running on Apple Silicon via UTM, a Splunk SIEM ingesting Windows Security logs from every host through Universal Forwarders, and a hybrid-identity bridge into Microsoft Entra ID — the foundation for simulating real attacks and writing the SPL detections that catch them.

Full Project Journal ↗

Lab Overview

The goal of this lab is to recreate a small enterprise network end-to-end and then defend it. Rather than spin up managed cloud services, I built the whole environment on a single Mac to prove the fundamentals: virtualization, Active Directory administration, log pipelines, and SIEM analysis. Once data is flowing, the same environment becomes a range for adversary simulation and detection engineering.

  • Virtualization layer: Windows Server 2022 (Domain Controller) and a Windows desktop client, both running as ARM VMs under UTM on Apple Silicon.
  • Active Directory: Promoted the server to a Domain Controller, then built out Organizational Units, users, and groups to mirror a real domain structure.
  • SIEM: Splunk running on the Mac as the indexer/receiver, with Splunk Universal Forwarders on each Windows host shipping WinEventLog:Security data over TCP 9997.
  • Hybrid identity: A free Microsoft Entra ID tenant connected to the on-prem directory via Entra Cloud Sync — the modern replacement for the decommissioned Entra Connect.

Getting Logs Into Splunk

With the domain running, the next milestone was wiring every Windows host into Splunk so security events could be searched in one place. Both Windows hosts now forward into the Mac-hosted Splunk receiver on port 9997:

  • Desktop client (DESKTOP-EL6QVN3): 3,624 events landing, with WinEventLog:Security visible — including EventCode 4624 (logon) and 4672 (special privileges assigned).
  • Domain Controller (WIN-83VFO41O2VB): 2,034 events landing and indexed.
  • Splunk receiver on TCP 9997 confirmed enabled and accepting forwarder connections.

Lessons Learned the Hard Way

The most valuable part of building infrastructure by hand is the troubleshooting. A few hard-won takeaways from getting the forwarding pipeline working:

  • A fresh Universal Forwarder ships nothing by default. I had to author an inputs.conf with [WinEventLog://...] stanzas on each host before any data flowed — the 9997 TCP connection alone carries no log data.
  • The Notepad trap (~1 hour lost on the DC): Notepad silently saved inputs.conf.txt while the real inputs.conf stayed empty, so the forwarder read nothing. The fix: always run Get-Content inputs.conf to confirm it has content before restarting.
  • Disk space matters: Splunk refuses to run searches when free disk drops below 5000 MB — had to clear space on the Mac.
  • PATH gotcha: splunk.exe isn't on PATH; restart with the full path — & "C:\Program Files\SplunkUniversalForwarder\bin\splunk.exe" restart.

Hybrid Identity with Entra ID

With logging in place, the lab grew toward hybrid identity — syncing the on-prem Active Directory into a cloud Microsoft Entra ID tenant.

  • Created a free Entra tenant ("ForMy Labs") via the M365 Business trial no-card route — the tenant directory is provisioned before the payment-card wall and persists afterward.
  • Chose Entra Cloud Sync (a lightweight provisioning agent) over the now-decommissioned classic Entra Connect MSI.
  • Blocker found & diagnosed: the Cloud Sync config page returned HTTP 401 — "Guest users are not allowed to configure sync." because the Gmail sign-in was a guest/external identity. The fix: create a cloud-only member admin, assign Global Administrator, and reconfigure sync as that account.

Roadmap

This lab is an in-progress build. With the domain, SIEM pipeline, and hybrid identity foundation in place, the remaining phases turn it into a full attack-and-detect range:

  • Finish Entra Cloud Sync with the cloud-only member admin account.
  • Stand up a Kali Linux attack box on the same network.
  • Run attack simulations against the domain (logon abuse, privilege escalation, lateral movement).
  • Write SPL detection rules in Splunk to catch each technique.
  • Build dashboards to visualize detections and host telemetry.
